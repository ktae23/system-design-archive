# 12주 학습 로드맵 (Roadmap)

> **대상**: 5년차 백엔드 개발자 — "경험은 있는데 언어가 부족한 성실한 실무형"
> **목표**: 트레이드오프를 근거로 말할 수 있는 시니어 후보로 12주 안에 전환
> **원칙**: (1) 이론 단독 주간 절대 금지 — 챌린지 안에서 이론이 호출된다 (2) 설계 문서는 매주 산출물이자 이력서 자양분 (3) 공부형이 아니라 실행형 — 완벽주의 차단, 판단 능력으로 측정

## Phase 1 — DB를 거점으로 "트레이드오프 언어" 장착 (Week 1~4)

### 목표
**가장 단단한 DB 영역에서 출발해 "Redis 박았어요"를 "Cache Aside로 갔고 Write Through는 ○○ 이유로 제외"로 바꾼다.**

### 해소할 격차
- **P0-1** 설계 판단을 트레이드오프 언어로 압축 (핵심)
- **P0-2** DB 동시성 시나리오를 엮어서 설명 (핵심)
- P1-3 실무 자료구조 감각 (Redis 자료형 초입)
- P0-3 레이어 거슬러 장애 추적 (운영 관점 초입, W3)

### 핵심 챌린지
1. **W1 — DB Anomaly 실습** (워밍업): MySQL 세션 2개 띄우고 Dirty/Non-repeatable/Phantom/Lost Update 4개를 직접 재현. **챌린지라기보다 "본인이 이미 잘 아는 영역"에서 손 감각 잡는 주간.** 실행형 페르소나의 초반 과부하 방지 목적.
2. **W2~W3 — 인기 상품 조회 API**: 10만 RPS read, 초당 100회 write. W2는 설계 v1(캐싱 전략 결정), W3는 v2(배포 중 무효화, Hot Key 감지, 커넥션 풀 등 운영 관점).
3. **W4 — 티켓 예매 시스템** (Phase 1 클라이맥스): 1000석에 10만 명, 좌석 중복 절대 금지. P0-2를 완성하는 주. 낙관/비관 락 선택, 결제 3~10초 동안 락 유지 불가능성, 큐 도입 결정까지.

### Phase 1 종료 시 "이걸 할 수 있어야 한다"

- [ ] **Lost Update와 Phantom Read를 서로 다른 시나리오로** 각각 설명하고 각각의 방어 방법 제시
- [ ] **Cache Aside / Write Through / Write Behind / Refresh Ahead 4개 패턴**을 "언제 어느 걸 쓰는지" 기준으로 구분
- [ ] **Cache Stampede 방어 3가지**(singleflight, randomized TTL, soft expiry)를 **각각 다른 상황**에서 쓰는 기준 설명
- [ ] 자기가 내린 설계 결정에 대해 **"왜 그걸 골랐는가"를 트레이드오프 2축**(예: Consistency vs Latency)으로 언어화
- [ ] 트레이드오프 언어 5축(Consistency vs Availability, Latency vs Throughput, Cost vs Complexity, Coupling vs Duplication, Fail-Fast vs Fail-Safe) 각각 **본인 설계 사례 1개씩 매핑**

### 면접 관점 기대 효과
> Phase 1이 끝나면 면접관은 이 사람을 이렇게 읽는다:
> **"DB 동시성에서 질문하면 교과서 답이 아니라 시나리오로 답이 나온다. 캐싱 관련 답변에 근거가 붙는다. 5년차 중위는 확실히 넘는다."**
>
> 아직 안 읽히는 것: 분산 환경에서의 실패 시나리오, 레이어 거슬러 진단 능력. 이건 Phase 2의 몫.

## Phase 2 — 레이어를 거슬러 + 분산 세계 입문 (Week 5~8)

### 목표
**"p99 튀어요" 한마디에 JVM → OS → 커널 → 네트워크 → DB 체크리스트가 자동으로 펼쳐진다. 그리고 서비스 간 실패 전파를 막는 방어막을 언어로 가진다.**

### 해소할 격차
- **P0-3** 레이어 거슬러 장애 추적 (핵심)
- **P1-1** 네트워킹 애플리케이션 레이어까지 연결
- **P1-2** OS 성능/장애 트리거 감각
- **P1-4** 분산 시스템 방어 패턴 (핵심)

### 핵심 챌린지
1. **W5~W6 — p99 SLO 200ms API**: 내부 의존성 3개 + 외부 결제 API(p99 500ms). W5는 latency budget·timeout 설계, W6은 Circuit Breaker·Bulkhead·Hedged Request로 방어막.
2. **W7 — MSA 10개 통신 레이어**: gRPC/REST 혼재, Connection Pool 사이즈 결정, Keep-Alive/LB Idle timeout 정렬.
3. **W8 — 주문→결제→재고 Saga**: 4단계 연쇄, PG는 2PC 참여 불가. Saga 2가지 구현(Choreography/Orchestration), Outbox 패턴, Idempotency Key 설계.

### Phase 2 종료 시 "이걸 할 수 있어야 한다"

- [ ] Resilience 5종(Timeout/Retry+Backoff+Jitter/Circuit Breaker/Bulkhead/Hedged)을 **각각 "쓰지 않으면 생기는 장애"와 짝**지어 설명
- [ ] **Fan-out 호출에서 p99가 왜 개별 호출 p99보다 악화되는지** 수학적 직관으로 설명
- [ ] HTTP/1.1 vs HTTP/2 vs HTTP/3의 HoL Blocking 차이, **gRPC가 HTTP/2 위에서 도는 이유** 설명
- [ ] Connection Pool 사이즈를 "경험상 200개"가 아니라 **Little's Law로 계산**
- [ ] Saga의 Choreography vs Orchestration을 **본인 도메인 예시로** 선택 근거 제시
- [ ] Idempotency Key 저장 위치 3곳(헤더/Gateway/DB) 장단점 + **TTL 산정 기준**

### 면접 관점 기대 효과
> Phase 2가 끝나면 면접관은 이렇게 읽는다:
> **"장애 상황을 던져도 당황하지 않는다. 레이어별 체크리스트가 있다. 분산 시스템 설계에서 '그냥 retry 걸어요' 수준이 아니라 방어 패턴 이름이 구체적으로 나온다. 시니어 후보 레인지 진입."**
>
> 아직 안 읽히는 것: 실무 자료구조 감각(특히 Redis 내부), 자기 결정을 공개 포트폴리오로 전시하는 능력. Phase 3의 몫.

## Phase 3 — 자료구조 감각 완성 + 이직 실행 (Week 9~12)

### 목표
**"Redis ZSET 쓰면 됩니다"에 "내부가 Skip List라서 O(log N + M)입니다"를 자연스럽게 붙인다. Phase 1~2의 설계 문서 자산을 이력서·포트폴리오로 외화(外化)해 실제 지원까지 실행한다.**

### 해소할 격차
- **P1-3** 실무 자료구조 감각 완성 (핵심)
- **P2-1** 알고리즘 면접 루틴 마무리
- 이직 준비 전 영역 (이력서, 포트폴리오, mock interview, 실제 지원)

### 핵심 챌린지
1. **W9 — 피드 서비스**: 1천만 사용자, 타임라인 + 랭킹 + 읽음 처리 + 중복 알림 방지. **Skip List / Bloom Filter / HyperLogLog / Bitmap**이 각각 필요해지는 지점으로 설계.
2. **W10 — Rate Limiter**: 시스템 디자인 레퍼토리 확장. Token Bucket / Leaky Bucket / Sliding Window (Log/Counter) 4가지 알고리즘을 자료구조와 함께 비교. 단일 서버 vs 분산 환경 차이.
3. **W11 — 이력서 대개조**: Phase 1~2 설계 문서 6개를 포트폴리오 5개로 압축. 경력기술서 STAR + 트레이드오프 언어. GitHub 공개.
4. **W12 — 모의 면접 총정리**: 시스템 디자인 2회 + 기술 꼬리질문 1회 + 행동 면접 1회. 이력서 v3 완성, 실제 지원 실행.

### Phase 3 종료 시 "이걸 할 수 있어야 한다"

- [ ] Redis 자료형 6종과 **내부 자료구조 매핑**(Skip List / HashTable / ListPack 등)을 근거로 자료형 선택
- [ ] Bloom Filter / HyperLogLog / Bitmap을 **"언제 정확한 자료구조 대신 확률적 자료구조로 가는가"** 기준으로 선택
- [ ] Rate Limiter 4가지 알고리즘의 **메모리 × 정확도 × 구현 복잡도** 매트릭스를 본인 언어로 설명
- [ ] 이력서 v3에서 모든 프로젝트가 **"문제 → 결정 → 근거 → 결과"** 구조로 서술
- [ ] 포트폴리오 5개 GitHub 공개 (각 README에 트레이드오프 섹션)
- [ ] Mock Interview에서 **"모르겠습니다"를 "여기까진 알고, 이 방향일 것 같습니다"로** 치환 성공

### 면접 관점 기대 효과
> Phase 3이 끝나면 면접관은 이렇게 읽는다:
> **"자료구조 선택에 근거가 있다. 공개 포트폴리오에 트레이드오프 섹션이 있다. 시니어로 볼만하다. 코드 품질 확인만 하면 된다."**
>
> 12주 전과의 결정적 차이: "경험 서술자"에서 **"결정 설명자"**로의 전환.

| 요일 | 활동 | 시간 |

### 반드시 지킬 3가지

1. **이론 먼저 읽지 않기** — 월요일 챌린지 브리핑 후에도 바로 이론 검색하지 않는다. 화요일 막힐 때까지 버틴다.
2. **설계 문서 품질 유지** — "같은 팀 동료가 읽어도 결정 근거가 이해된다" 기준. Phase 3 포트폴리오 원재료.
3. **"말문 막힌 지점" 매주 기록** — 금요일 회고에서 답 못 한 주제 리스트화. 다음 주 학습 방향의 실마리.

## 성공/실패 시그널

### Phase 1 체크포인트 (W4 종료 시)

**잘 가고 있음**
- 설계 문서 2개 완성, 공개해도 부끄럽지 않은 품질
- DB Anomaly 4개를 세션 두 개로 직접 재현한 노트 존재
- 매주 회고에 "이번 주에 몰랐던 것" 리스트가 **3개 이상** 있음 (= 자기 인식이 살아있음)
- 트레이드오프 축 5개 중 최소 3개를 본인 사례로 매핑 완료

**조정 필요**
- 회고에 "이번 주에 몰랐던 것"이 비어있음 → **모르는 걸 못 알아차리는 상태**. 멘토 리뷰 주기 단축
- 이론을 월요일에 먼저 읽고 있음 → 역학습이 깨지는 중. 월~화는 설계만, 수요일부터 이론 규칙 재고정
- 설계 문서가 "Redis 썼어요"로 끝남 → 트레이드오프 언어 강제 체크리스트 도입

### Phase 2 체크포인트 (W8 종료 시)

**잘 가고 있음**
- p99 SLO API의 Circuit Breaker 미니 데모가 **실제로 돌고 있음** (코드 자산)
- 본인 현업 서비스의 커넥션 풀 사이즈를 Little's Law로 재계산해본 문서 있음
- "장애 플레이북 10개" 표 작성 완료, 각 항목에 레이어 태깅

**조정 필요**
- 이론을 **깊게** 파고 있음 (예: JVM GC 알고리즘 비교 한 주 내내) → 역학습 궤도 이탈. 챌린지로 복귀
- 설계 문서에 **동일한 패턴 이름**만 반복 (예: "Circuit Breaker 쓴다"가 모든 문서에) → 패턴 적용 맥락 분화 필요
- Saga를 Orchestration만 쓰거나 Choreography만 쓰고 있음 → 두 방식 모두 본인 예시로 쓰게 과제 조정

### Phase 3 체크포인트 (W12 종료 시)

**잘 가고 있음**
- Mock 4회 로그에서 **"모르겠습니다" 횟수가 감소** 추세
- GitHub 포트폴리오 공개 2개 이상
- 실제 지원 3곳 이상 실행
- 12주 회고 문서에 "Before/After 역량 매트릭스"가 구체적 문장으로 채워짐

**조정 필요**
- Mock 결과가 Phase 1과 차이 없음 → 설계 문서는 쌓였으나 **말하기 근육**이 부족. Mock 횟수 증가 또는 녹화 후 자가 분석
- 포트폴리오 README에 **트레이드오프 섹션 누락** → 이력서 자양분 역할 실패. 재작성 필수
- 이직 지원 미실행 → 완벽주의 발동 신호. **"충분히 준비된 날은 안 온다"** 원칙으로 강제 제출

## 12주 후 완성 이미지 (3가지 관점 스냅샷)

### 1) 이력서 관점 — "Before → After"

## DB 중급 중위라는 자산을 어떻게 활용하는가

이 사람의 5개 영역 중 DB가 가장 단단하다. 로드맵은 이 사실을 적극 활용한다:

1. **W1을 DB 실습으로 배치** — "가장 잘하는 것"에서 손을 풀며 페이스를 잡는다. 실행형 페르소나는 초반 과부하가 가장 큰 이탈 리스크
2. **W2~W3 캐싱 챌린지에서도 DB 관점 질문으로 연결** — "캐시 무효화 vs DB 부하" 같은 식으로 DB 감각을 캐싱 판단의 레버리지로 사용
3. **W4 티켓 예매에서 DB 동시성을 완성** — Phase 1 클라이맥스에서 가장 단단한 영역을 **'시스템 디자인 언어'와 결합**. "MVCC는 안다" → "MVCC를 설계 판단의 근거로 쓴다"
4. **이후 Phase에서도 DB를 기준점으로** — p99 SLO API에서 "DB slow query도 p99의 한 원인", Saga에서 "Outbox는 DB 트랜잭션으로 이벤트를 묶는 패턴"처럼 DB 감각이 다른 영역을 끌어당기는 축이 됨

## 커리어 트랙 (이력서/포트폴리오)이 Phase 3에 몰리지 않는 이유

**이직 준비는 Week 11~12에 집중되지만, 그 준비는 Week 1부터 이미 시작된다.**

- 매주 작성되는 **설계 문서**가 곧 포트폴리오의 원재료
- 매주 회고에서 **"말문 막힌 지점 기록"**이 Mock Interview 대비 자료
- Phase 1~2 동안 축적되는 **"기술 결정의 근거"**가 경력기술서의 Action → Result 연결고리

이 구조 때문에 Week 11~12에 "갑자기 이력서를 써야 하는" 부담이 없다. **"지금까지 쌓은 자산을 edit만 하면 되는" 단계**가 된다. 이게 이력서 대개조 주간을 2주(W11~W12)로 압축할 수 있는 이유.
