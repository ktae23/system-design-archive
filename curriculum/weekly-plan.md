## Week 1 (Phase 1) — DB Anomaly 실습

**이번 주 챌린지**: MySQL 세션 2개로 Transaction Anomaly 4종을 직접 재현하고 방어하기
- 요구사항: Dirty Read / Non-repeatable Read / Phantom Read / Lost Update를 각각 다른 격리 수준에서 재현 → 방어
- 트래픽/제약: 로컬 MySQL 1대 + 터미널 2개. **이번 주는 의도적으로 가볍게**. 워밍업과 페이스 잡기가 목적

**해소되는 격차**: P0-2 DB 동시성 엮기 (initial — Phase 1 클라이맥스 W4에서 완성)

**챌린지 진행 중 "막히는 지점"에서 들여올 이론 2~3개**
- **Isolation Level 4종** — "왜 이 레벨에서 이 현상이 안 나오는가?"에서 막힐 때 호출
- **MVCC 스냅샷 시맨틱** — MySQL REPEATABLE READ가 Phantom까지 막는 이유(Gap Lock 개입)에서 막힐 때
- **Lost Update 방어 2가지** — Pessimistic(`SELECT FOR UPDATE`) vs Optimistic(Version Column) 선택 기준

**주간 산출물**
- `system-design/w01-db-anomaly-lab.md` — Anomaly 4개 재현 SQL + 방어 SQL + 내 해설 (이력서 재료로 축적)
- `curriculum/retrospective/w01.md` — 회고 메모

**회고 질문 3개 (금요일 자답)**
1. REPEATABLE READ에서 Phantom이 MySQL에서는 안 나오고 PostgreSQL에서는 경우에 따라 나온다 — 이 차이를 한 문단으로 설명할 수 있는가?
2. Lost Update를 발견했을 때 Pessimistic vs Optimistic을 고르는 **내 판단 기준**을 두 문장으로 쓸 수 있는가?
3. 다음 주 "인기 상품 조회 API" 설계에서 **이번 주에 얻은 DB 감각**이 어떻게 쓰일지 1가지 예측은?

**백그라운드 과제**: LeetCode 2문제 (HashMap/Array 기초 — 감각 깨우기)

**체크포인트**
- Anomaly 4개 재현 SQL이 각각 실행 가능한 상태 + 방어 결과 기록
- "MVCC와 Lock은 다르다"를 **자기 언어로** 문장 3개 안에 정리할 수 있다 (판단 능력: 주어진 시나리오에서 둘 중 어느 관점으로 볼지 선택 가능)

**면접관 시점 코멘트**
> 이 주가 끝나면 "격리 수준 차이?" 질문에 교과서 정의 대신 **"세션 두 개 띄워보니 이랬어요"** 경험 기반 답이 나온다. 시니어 인상의 출발점.

## Week 2 (Phase 1) — 인기 상품 조회 API (1차)

**이번 주 챌린지**: 이커머스 인기 상품 TOP 100 조회 API 설계 v1 (캐싱 전략 결정)
- 요구사항: read 10만 RPS, write(재고/가격) 초당 100회. MySQL 마스터 1 + 리플리카 2. Redis 사용 가능. 정합성 허용 윈도우는 PM과 협의 가능(이 협의 자체를 문서화 대상)
- 트래픽/제약: 이번 주 초점은 "4가지 캐싱 패턴 중 **어떤 걸 왜 고르는가**"만 확실히

**해소되는 격차**: P0-1 트레이드오프 언어화 (핵심 시작), P1-3 자료구조 감각 (Redis 초입)

**챌린지 진행 중 "막히는 지점"에서 들여올 이론 2~3개**
- **Caching Patterns 4종**(Cache Aside / Write Through / Write Behind / Refresh Ahead) — "어떻게 갱신하지?"에서 막힐 때
- **Consistency 스펙트럼**(Strong / Bounded Staleness / Eventual) — "정합성 허용 윈도우"를 PM 언어로 정의하려다 막힐 때
- **TTL 설계 감각** — "TTL을 왜 5분? 1분? 30초?"의 근거를 찾으려다 막힐 때

**주간 산출물**
- `system-design/w02-hot-product-api-v1.md` — 요구사항 + 4가지 패턴 비교 + 선택 근거 + TTL 결정 근거
- `curriculum/retrospective/w02.md`

**회고 질문 3개**
1. Cache Aside와 Write Through의 선택 기준을 **"읽기/쓰기 비율"과 "정합성 허용 윈도우"** 두 축으로 설명할 수 있는가?
2. 이번 설계에서 내가 "프로덕트와 협의 가능"한 지점을 몇 개 식별했는가? 그 협의를 **설계 문서에 어떻게 담았는가**?
3. 다음 주 "운영 관점 확장"에서 **이번 주 설계가 깨질 가능성**이 가장 큰 지점은 어디라고 예측하는가?

**백그라운드 과제**: LeetCode 3문제 (HashMap / TwoSum 계열 — 자료구조 감각)

**체크포인트**
- "Cache Aside로 갔고 Write Through는 ○○ 이유로 제외했습니다"를 **같은 팀 동료가 납득**하는 수준의 문단으로 쓸 수 있음 (판단 능력: 조건을 바꿔 "write가 지배적이면 선택이 어떻게 달라지나"에 즉답 가능)
- 정합성 허용 윈도우를 **숫자로** 정의 (예: "30초 이내 재고 오차 허용")

**면접관 시점 코멘트**
> 이 주가 끝나면 "캐시 도입 경험?" 대신 **"○○ 조건이라 Cache Aside, Write Through는 ◇◇ 리스크로 제외"** 근거 구조가 나온다. 면접관이 꼬리질문을 더 던지게 됨 — 좋은 신호.

## Week 4 (Phase 1 클라이맥스) — 티켓 예매 시스템

**이번 주 챌린지**: 공연 티켓 예매 시스템 설계 — 동시성 지옥의 종합판
- 요구사항: 좌석 1,000석, 오픈 순간 동시 접속 10만 명, 좌석 **중복 예매 절대 금지**, **초과 판매 절대 금지**, 결제는 외부 PG(3~10초, 실패 가능)
- 트래픽/제약: 이번 주는 Phase 1의 모든 학습(DB 동시성 + 캐싱 + 트레이드오프 언어)이 하나로 엮이는 주

**해소되는 격차**: P0-2 완성 (DB 동시성 엮기), P0-1 강화

**챌린지 진행 중 "막히는 지점"에서 들여올 이론 2~3개**
- **Optimistic vs Pessimistic Locking** — "동시에 같은 좌석 예매 막기"에서 막힐 때
- **Next-Key Lock / Gap Lock** — `SELECT FOR UPDATE` 했더니 deadlock, 왜? 에서 막힐 때
- **"락이냐 큐냐" 설계 전환** — 10만 명을 DB 락으로 받을 수 없다는 자각에서, 큐 도입의 이론적 근거

**주간 산출물**
- `system-design/w04-ticket-booking.md` — 예매 시퀀스 + 동시성 방어 전략 + "락 vs 큐" 선택 근거 + 결제 3~10초 처리 방식
- `system-design/w04-concurrency-lab.md` — W1 랩의 확장: Lost Update를 비관/낙관 두 가지로 방어하는 코드
- `curriculum/retrospective/w04.md` + **Phase 1 통합 회고**

**회고 질문 3개**
1. 결제 3~10초 동안 DB 락을 쥐고 있으면 안 되는 이유를 **"가용성" 관점**과 **"deadlock 확률" 관점** 두 가지로 설명할 수 있는가?
2. 본인이 고른 "낙관 락 vs 비관 락 vs 대기열 큐"의 조합을 **어떤 트레이드오프 축**(Consistency vs Throughput 등)으로 정당화하는가?
3. Phase 1 4주를 마친 지금, **가장 크게 바뀐 것**이 "지식"인지 "언어"인지 "자신감"인지 — 하나를 고르면?

**백그라운드 과제**: LeetCode 3문제 (LinkedList / Queue — 대기열 감각)

**체크포인트**
- "좌석 중복 예매 절대 금지"를 **3가지 레이어**(DB Constraint / 비관 락 / 대기열)에서 각각 어떻게 막는지 표로 정리 (판단 능력: 각 레이어의 실패를 가정했을 때 다른 레이어가 어떻게 커버하는지 설명 가능)
- W1~W4 설계 문서 3개를 한 번에 읽었을 때 **"4주 동안 성장했구나"**가 느껴지는 수준

**면접관 시점 코멘트**
> "티켓 예매처럼 경합 심한 시스템?" 같은 **고전 시스템 디자인 질문**에서 책 암기 답이 아니라 **내 실험과 결정의 언어**로 답이 나온다. Phase 1 총정리이자 Phase 2 티켓.

## Week 5 (Phase 2) — p99 SLO 200ms API (1차, latency budget)

**이번 주 챌린지**: 결제 확인 API 설계 — 꼬리 지연과의 전쟁 v1
- 요구사항: 내부 의존성 3개(user/order/fraud) + 외부 결제 API 1개(p99 500ms / 평균 150ms / 실패율 0.5%). SLO **p99 200ms**
- 트래픽/제약: 외부 API p99(500ms)가 우리 SLO(200ms)를 이미 초과. 어떻게 지킬 것인가가 설계의 전부

**해소되는 격차**: P0-3 레이어 거슬러 (핵심), P1-2 OS 감각 (초입)

**챌린지 진행 중 "막히는 지점"에서 들여올 이론 2~3개**
- **Latency Budget** — "각 단계에 시간을 어떻게 쪼개지?"에서 막힐 때. Critical Path 식별
- **Tail Latency Amplification**(Jeff Dean "Tail at Scale") — "Fan-out 4개 호출의 p99가 왜 개별보다 더 나쁜가?"에서 막힐 때
- **Timeout 원칙** — "Timeout을 몇 초로?"의 근거(퍼센타일 기반 + safety margin)

**주간 산출물**
- `system-design/w05-p99-slo-v1.md` — 호출 시퀀스 + **latency budget 표**(각 단계 예산) + Timeout 값 근거
- `curriculum/retrospective/w05.md`

**회고 질문 3개**
1. 평균 150ms인 외부 API에 **Timeout을 얼마로** 설정할 것인가? **p50이 아니라 p99 기반으로 잡아야 하는 이유**를 한 문단으로?
2. Fan-out 호출이 개별 호출보다 p99가 나빠지는 현상을 **확률 직관**으로 설명 가능한가? (각 호출 p99 확률 1%일 때 4개 중 하나라도 걸릴 확률?)
3. 다음 주 "장애 방어"로 가기 전, **지금 설계의 가장 큰 취약점**을 하나 꼽으면?

**백그라운드 과제**: LeetCode 3문제 (Binary Search — 퍼센타일 감각)

**체크포인트**
- Latency Budget 표가 **p50 / p95 / p99 / timeout** 4열로 정리됨
- 외부 API p99 500ms를 우리 SLO 200ms에 넣는 **최소 2가지 방법** 제시 (판단 능력: 조건을 바꿔 "외부 API의 p99가 1초로 악화되면?" 즉답 가능)

**면접관 시점 코멘트**
> "p99 튀는 이유?"에 **"Fan-out에서의 tail latency amplification"** 같은 구체적 개념어가 나온다. "숫자로 설계하는 사람" 인상.

## Week 6 (Phase 2) — p99 SLO 200ms API (2차, 장애 방어)

**이번 주 챌린지**: W5 설계에 **장애 방어 패턴** 추가 — Resilience의 언어 장착
- 요구사항: 외부 결제 API가 **갑자기 느려지거나 죽을 때** 우리 서비스도 같이 죽지 않을 것. **p99 SLO 유지하면서** 방어
- 트래픽/제약: W5 설계 그대로 + 장애 패턴 5종 중 **필요한 것만** 선별 적용

**해소되는 격차**: P0-3 완성, P1-2 OS 감각, P1-4 분산 방어 (초입)

**챌린지 진행 중 "막히는 지점"에서 들여올 이론 2~3개**
- **Circuit Breaker 3-state**(Closed / Open / Half-Open) — "외부 API 죽었을 때 우리도 같이 죽음"에서 막힐 때
- **Bulkhead Pattern** — "외부 API 호출 스레드가 우리 전체 스레드풀을 잡아먹음"에서 막힐 때
- **Retry with Exponential Backoff + Jitter** — "재시도 폭탄으로 외부 API와 우리가 동반 사망"에서 막힐 때

**주간 산출물**
- `system-design/w06-p99-slo-v2.md` — W5 + Resilience 5종 적용 근거 표(각각 "썼다/안 썼다 + 근거")
- **Circuit Breaker 미니 데모** (Resilience4j / Polly / hystrix-go 등 택1) — 실제 코드 자산

## Week 8 (Phase 2 마무리) — 주문→결제→재고 Saga

**이번 주 챌린지**: 이커머스 주문 플로우 — 분산 트랜잭션 설계
- 요구사항: 주문 → 결제(외부 PG) → 재고 → 배송. **2PC 금지**. 중간 실패 시 롤백, 메시지 중복 처리로 인한 2번 결제/차감 방지
- 트래픽/제약: Kafka 사용 가능. DB는 각 서비스별 독립

**해소되는 격차**: P1-4 분산 방어 (완성)

**챌린지 진행 중 "막히는 지점"에서 들여올 이론 2~3개**
- **Saga Pattern (Choreography vs Orchestration)** — "2PC 말고 어떻게?"에서 막힐 때. 보상 트랜잭션 설계 원칙
- **Idempotency Key 설계** — "메시지 중복으로 2번 결제"를 막으려다 막힐 때. 저장 위치 3곳(헤더/Gateway/서비스 DB), TTL 산정
- **Outbox Pattern + CDC** — "DB 커밋 직후 이벤트 발행 실패 = 이벤트 유실"에서 막힐 때

**주간 산출물**
- `system-design/w08-saga.md` — 4단계 Saga 다이어그램 **2가지 버전**(Choreography / Orchestration) + 선택 근거
- **Outbox 미니 프로토타입** — 단일 DB + Kafka 로컬, 언어 자유
- `curriculum/retrospective/w08.md` + **Phase 2 통합 회고**

**회고 질문 3개**
1. Choreography와 Orchestration 중 본인이 고른 방식의 근거를 **"서비스 수 × 커플링 허용도"**로 설명 가능한가?
2. "at-least-once delivery + idempotent consumer = effectively exactly-once"의 의미를 **각 구성요소의 역할**로 풀어 설명 가능한가?
3. Idempotency Key의 TTL을 **얼마로** 잡을 것인가? 너무 짧을 때 / 너무 길 때 각각의 문제는?

**백그라운드 과제**: LeetCode 3문제 (Topological Sort / DAG — Saga 단계 감각)

**체크포인트**
- Saga 다이어그램 2가지 버전 모두 **보상 트랜잭션 경로 포함** (판단 능력: 서비스가 15개로 확장되면 두 방식 선택이 어떻게 달라지는지 즉답)
- Outbox 프로토타입이 **DB 커밋 + 이벤트 발행의 원자성**을 실증
- Phase 2 4주 돌아봤을 때 **"분산 시스템의 어떤 실패 모드"가 여전히 낯선지** 1개 이상 식별

**면접관 시점 코멘트**
> "분산 트랜잭션 어떻게 하시나요?"에 "2PC의 한계 → Saga 2가지 → Idempotency + Outbox"가 유기적으로 연결된다. 시니어 후보 레인지 확실히 진입.

## Week 9 (Phase 3) — 피드 서비스

**이번 주 챌린지**: 소셜 피드 서비스 설계 — 자료구조 감각의 총동원

- 트래픽/제약: Redis 사용 가능. 각 기능마다 "어떤 자료구조가 왜 적합한가"를 명시

- `system-design/w09-bloom-calc.md` — Bloom Filter 수동 계산(알림 1억 건, FPR 1% → 비트 수 / 해시 함수 수)

**회고 질문 3개**

2. Bloom Filter를 "정확한 Set" 대신 고르는 **손익 공식**(메모리 절약 vs False Positive 허용 비용)을 본인 도메인으로 풀 수 있는가?

**백그라운드 과제**: LeetCode 3문제 (Heap / Priority Queue — 랭킹 감각)

**체크포인트**

**면접관 시점 코멘트**
> "실시간 랭킹 어떻게 설계?"에 "Redis ZSET 씁니다 — 내부가 Skip List라서 O(log N + M)입니다"가 한 호흡으로 나온다. 자료구조를 **"면접 암기"가 아니라 "설계 근거"**로 쓰는 레벨.

## 운영 원칙 (12주 내내 고정)

1. **월요일 이론 금지** — 챌린지 브리핑 후 바로 설계 시작. 이론은 화요일 이후 막힌 지점에서만
2. **설계 문서 품질** — 매주 금요일 공개 가능 수준으로 완성. Phase 3 포트폴리오 원재료
3. **회고 질문 3개 무조건 답변** — "몰랐던 것"을 매주 발견. 안 발견하면 자기 인식이 죽어가는 신호
4. **체크포인트는 "판단 능력"** — "설명할 수 있다"가 아니라 "조건이 바뀌면 결정도 바꿀 수 있다"로 기술
5. **Week 1 가볍게** — 실행형 페르소나의 12주 완주율을 결정하는 변수. 초반 과부하 = 이탈
6. **알고리즘 총량 경쟁 금지** — 매주 2~3문제, W10~W12는 2문제. 꾸준함만 본다
7. **면접관 시점 코멘트 매주 확인** — "이 주가 끝나면 면접관에게 어떻게 읽힐지"를 자기 거울로 사용
