## 1단계 Q1 — "중복 주문의 규모를 어떻게 파악하나?"

### 내 첫 답변
> 주문에는 멱등성을 고려한 주문 키를 사용할텐데 중복이라함은 같은 멱등키가 2개 이상 DB에 존재한다는것(유니크가 아니라는 의미)인지, 주문키는 멱등키지만 완전히 동일한 주문건이 근시간(1~3초 이내)에 들어간건지 확인이 필요합니다. DB와 단일 인스턴스를 사용한다면 주문 요청 후 오래 걸려서 새로고침을 하거나 프론트에서 커넥션 타임아웃 후 다시 요청을 하는 등의 문제일 가능성이 있음.

### 멘토 피드백

**잘한 것**
- "중복의 정의"를 먼저 짚음 → 면접 감점 함정(솔루션부터 말하기) 회피
- 시나리오 (A) 동일 멱등키 다중 존재 / (B) 다른 멱등키 + 근시간 중복으로 분기
- 1단계에서 2단계 원인 추론까지 연결 (지연 → 타임아웃 → 재시도)

**보강**
- 스펙(Redis 없음, 큐 없음, 모놀리식)을 보면 **멱등키 인프라 자체가 부재**할 가능성이 높음 → 시나리오 **(C) 멱등키 자체 부재**를 추가로 열어둬야 함
- 시니어 습관: *"X라고 가정하면…"* 뒤에 *"X가 없다면…"*을 같이 붙이기

## 1단계 Q1-1 — "근시간(1~3초) 근거는?"

### 내 답변
> 1~3초는 결제 api 타임아웃 값이고 보통 사람이 길다고 느껴지는게 3초까지기 때문에 1~3초면 정말 다른 주문이라기보다는 중복이라고 볼수 있음

### 멘토 피드백
- ✅ "사람이 길다고 느끼는 3초" 앵커 정확 (Nielsen Response Time Limits 2~3초와 일치)
- ⚠️ "결제 API 타임아웃 3초"는 부정확 — 실제 PG 권장은 **15~30초** (3DS/카드사 조회 타니까)
- 중복 트리거 앵커 4가지:
  - 유저 인내 한계: 2~3초
  - Nginx `proxy_read_timeout`: 60초
  - ALB/ELB idle timeout: 60초
  - PG 타임아웃: 15~30초
- 1~3초 창은 **유저 재시도만** 잡고 **인프라 재시도는 놓침**

## 1단계 Q1-2 — "중복 판별 의사 SQL + 정상 주문과 구분"

### 내 첫 답변 (문법 오류 포함)
```sql
SELECT o1.id, o2.id FROM orders o1
INNER JOIN orders o2 ON o1.user_id = o2.user_id AND o1.product_id = o2.product_id
WHERE TIME_SUB(o1.created_at, o2.created_at, SECOND) <= 3;
```

### 멘토 지적
- 자기 자신과의 조인 허용 (`o1.id = o2.id`도 통과)
- `TIME_SUB` 함수 오용 (의도는 `TIMESTAMPDIFF`)
- 시간 부호 문제 (`ABS()` 필요)
- **정상 주문(같은 상품 여러 개 따로 주문) 구분 조건 누락**

### 내 정정본
```sql
SELECT o1.id, o2.id FROM orders o1
INNER JOIN orders o2 ON o1.user_id = o2.user_id AND o1.product_id = o2.product_id
WHERE TIMESTAMPDIFF(o1.created_at, o2.created_at, SECOND) <= 3
  AND o1.id < o2.id
  AND o1.cart_id = o2.cart_id
  AND o1.quantity = o2.quantity
  AND o1.total_price = o2.total_price;
```

### 멘토 피드백
- ✅ 구분 조건 3개(`cart_id`, `quantity`, `total_price`) — 제공된 힌트보다 더 나아감
- ⚠️ `TIMESTAMPDIFF(unit, dt1, dt2)` 인자 순서 여전히 틀림 (단위가 **첫 번째** 인자)
- ⚠️ `cart_id`는 강한 가정 — 즉시 구매 플로우의 NULL 케이스 놓침
- 💡 멱등키 컬럼이 있다면 **`GROUP BY idempotency_key HAVING COUNT(*) > 1`로 시나리오 (A)부터** 먼저 잡는 게 진단 순서상 맞음

### 멘토 확정본
```sql
SELECT o1.id AS orig_id, o2.id AS dup_id, o1.created_at, o2.created_at
FROM orders o1
INNER JOIN orders o2
  ON o1.user_id = o2.user_id
 AND o1.product_id = o2.product_id
 AND o1.id < o2.id
WHERE o1.created_at > NOW() - INTERVAL 1 HOUR                     -- 폭증 시점만
  AND ABS(TIMESTAMPDIFF(SECOND, o1.created_at, o2.created_at)) <= 3
  AND (o1.cart_id = o2.cart_id OR o1.cart_id IS NULL)             -- 즉시 구매 케이스 포함
  AND o1.quantity = o2.quantity
  AND o1.total_price = o2.total_price;
```

**실무 포인트**: self-join은 큰 테이블에 부하 큼 → 시간 범위 한정 + `(user_id, created_at)` 인덱스 필수.

## 1단계 Q1-3 — "멱등키 인프라 유무 확인 3곳"

### 내 첫 답변
> 테이블 DDL에 idepotency_key 컬럼 또는 UUID 형식의 order_id 존재 확인 or 화면에 난수 주문번호 존재 여부 확인

### 멘토 지적 — 개념 혼동
- `order_id` ≠ `idempotency_key`
  - `order_id`: **서버 생성** PK, 주문 식별자
  - `idempotency_key`: **클라이언트 생성**, 헤더로 전달, **"같은 요청의 중복 실행 차단"** 장치
- 놓친 체크포인트: 서버 controller 입구 코드, 실제 네트워크 요청 헤더, API 문서 (Swagger/README)

### 내 재답변
> order_id는 sequence or auto increment 등 순차 id일 경우가 많아 정렬 이점이 높기 때문에 uuid를 pk로 잡지는 않으니까. idempotency_key는 유일 식별을 위해 사용하는 키로 order_id처럼 order의 식별자(pk)는 아니야

### 멘토 피드백
- ✅ 외형 차이(PK 여부) 정확
- ⚠️ **"왜 존재하는가"가 빠짐** — 멱등키의 본질은 "중복 실행 방지"
- 시니어 한 문장 정답:
  > `order_id`는 주문을 식별하는 서버 내부 PK(서버 생성)이고, `idempotency_key`는 **"같은 요청의 중복 실행"을 서버가 감지**하기 위해 **클라이언트가 생성해서 헤더로 전달**하는 외부 키. 서버는 같은 키 재요청 시 **로직 재실행 없이 이전 응답을 그대로 반환**.

### 파생 인사이트 (2단계 Q4 예고편)
**"UNIQUE 제약만으로 멱등성 보장 가능?"** → **아니오**.
UNIQUE는 2번째 INSERT에 에러만 낼 뿐, 클라이언트는 "이미 처리됐는지 / 실패했는지"를 구분 못 함 → 재시도 반복. 멱등키는 **"처리됨 + 같은 결과 반환"**으로 클라이언트가 안전하게 재시도 가능.

## 1단계 Q2 — "서버 상태 모니터링 지표"

### 내 답변
> CPU가 높다는것은 유휴 스레드가 없다는 의미이고, 대부분의 요청은 DB를 호출하기 때문에 DB 커넥션도 전부 사용중일거야.
> 때문에 중복 주문이 발생하는 것은 커넥션 타임 아웃으로 인해 요청이 중복 실행되는 것으로 추정이 돼
>
> 커넥션 타임아웃이 발생하고 있고 이로 인한 재요청이 발생하고 있는지 확인해야 돼
>
> apm이 있다면 응답 지연율, db 커넥션 풀, 데드락 을 보고 없다면 DB 프로세스 리스트를 확인해봐야 할것 같아

### 멘토 피드백

**결정적으로 잘한 것**
- **가설 → 검증 구조** (주니어는 지표 나열, 시니어는 가설 기반)
- **레이어 체인 추적**: CPU ↑ → 유휴 스레드 ↓ → DB 커넥션 풀 ↓ → 타임아웃 → 재시도 → 중복
- **P0-3 (레이어 거슬러 추적) 격차에서 첫 성과** — 진단 당시 "아직 약하다"고 본 부분이 Q2에서 실제로 구현됨

**보강 5가지**

1. **논리 구멍**: 타임아웃이 일어나는 "어디"가 흐림 → **HTTP/클라이언트 레이어**에서 타임아웃이지 DB가 아님
2. **각 지표에 "뭘 보려고?" 붙이기**:
   - p99 응답 지연 → *"latency 폭증이 재시도 유발?"*
   - DB 풀 active/max → *"풀 고갈이 지연 원인?"*
   - 롱런닝 트랜잭션 → *"블로킹으로 느려지나?"*
3. **중복 주문 특화 지표 4개 누락**:
   - 주문 생성률 이상 (트래픽 100배인데 주문 200~300배면 중복 포함 신호)
   - 응답 상태 코드 분포 (499/502 증가 = 재시도 직전 신호)
   - 분당 같은 user_id의 `POST /api/orders` 횟수
   - Nginx access log tail (처리 시간 컬럼)
4. **"데드락"은 이 시나리오에 빗나감** — 동기 INSERT 폭주에선 **락 대기 + gap lock + row lock contention**이 더 정확한 키워드. 데드락 꺼내면 *"어떤 쿼리가 데드락을?"*에 말문 막힘
5. **APM 없을 때 루틴 확장**:
   ```bash
   mysql> SHOW PROCESSLIST;
   mysql> SHOW ENGINE INNODB STATUS\G
   mysql> SHOW STATUS LIKE 'Threads_%';
   tail -f /var/log/nginx/access.log | awk '$7=="/api/orders"'
   vmstat 1                                        # 스냅샷 말고 추이
   netstat -an | awk '/:80/ {print $6}' | sort | uniq -c
   ```

## 1단계 Q2-후속 — "실시간 탐지 알람 설계"

### 내 답변
> 스레드풀 70, 90, 100퍼센트 사용중인 채로 30초 이상 유지시 경고

### 멘토 피드백

**시니어 감각 2개가 이미 박혀 있음**
- **다단계 임계값** (3-tier escalation) — 단일 임계값 안 쓰는 건 운영 경험 증거
- **"30초 이상 지속"** 조건 — 순간 스파이크 false positive 방지

**보강**
1. **알람 3종 체인 완성 필요**: "원인 근접" + "증상" + "현상 직접"
   - [원인] DB 커넥션 풀 active/max ≥ 90% 30초 지속
   - [증상] 스레드풀 70/90/100% 30초 지속 (네 답)
   - [현상] **분당 same user의 `POST /api/orders` ≥ 3회 발생 건수 ≥ 10건/1min** ← 중복 주문 실시간 탐지의 핵심
2. **동적 기준 미사용** — 고정값 70%는 정상 피크 시간대에도 터짐. baseline 대비 기준 섞기:
   - 일주일 전 같은 시간대 대비 200% 초과
   - p99 응답 시간 baseline의 3배 초과 + 30초 지속
3. **각 단계별 액션/Severity/Runbook 정의** — 알람이 살아남으려면 필수:
   | 임계 | Severity | 액션 |
   |---|---|---|
   | 70% | P3 (Slack) | 오토스케일링 검토, 런북 확인 |
   | 90% | P2 (담당자) | 풀 증설 / Rate Limit 조이기 |
   | 100% | P1 (온콜) | 긴급 모드, 서킷 브레이커 오픈 |

## 1단계 Q3 — "지금 당장" 막는 인프라 조치

### 내 답변
> AWS와 같은 클라우드 사용중이라면 API 게이트웨이 큐잉을 활성화해서 속도 제한을 걸어서 우리 서버 처리량을 견딜수 있도록 요청을 처리할 수 있는 시간을 벌어줍니다. 중복 제어가 문제기 때문에 완전히 동일한 주문건이라고 볼 수 있는 기준, 카트 아이디, 총액, 품목 등을 유니크키를 걸어 일시적으로 중복 주문을 하지 못하도록 막습니다. 이 방식이 주문 자체를 막는것보다는 매출 손실을 최소화하면서 중복건을 제어할 수 있습니다. 다만 유니크키를 거는 동안 메타데이터락이 걸려 행잉된 주문건에 대한 커넥션이 전부 드랍될 수 있고 db cpu 사용율이 더 오를 수 있기 때문에 커넥션이 중단되더라도 지장이 없는지 검토가 필요합니다.
>
> **실행 순서**: 1) 주문 버튼 일시 비활성화  2) 커넥션 행잉 해소 후 유니크 키 설정  3) API Gateway queue 설정  4) 주문 버튼 활성화

### 잘한 것 (P0-1 훈련 진척 신호)

- **트레이드오프를 문장으로 표현하기 시작** — *"매출 손실 최소화 vs 차단 강도"* 비교축이 답변에 등장. 진단 당시 "트레이드오프 언어 부재"라고 본 P0-1 항목에서 처음으로 직접 출현.
- **DDL 부작용을 인식** — 유니크 키 추가 → 메타데이터 락 → 커넥션 드랍 가능성 언급. 운영 DDL 한 번이라도 다뤄본 사람만 나오는 문장이야 (signal of operational scar tissue).
- **단계화된 실행 순서** — "버튼 OFF → 작업 → ON". 시니어가 런북 짤 때 쓰는 골격과 동일.

### 시니어 관점 보강 6가지

**1. ALTER TABLE은 지금 상황에서 "가용성 자살"**
이미 행잉된 커넥션이 풀을 점유 중인 상태에서 `ALTER TABLE ... ADD UNIQUE`를 치면, MySQL의 **MDL(Metadata Lock)**이 기존 트랜잭션 종료를 기다리며 대기 큐에 쌓이고 — 그 뒤에 들어오는 **모든 SELECT조차 MDL 대기로 멈춰버려**. 이게 MySQL의 가장 악명 높은 함정 (silent stampede). 운영 중인 큰 테이블에 유니크 키 거는 건 `pt-online-schema-change`나 `gh-ost` 같은 온라인 스키마 변경 도구를 써야 해. 5분 안에 누를 버튼이 절대 아냐.

**2. "cart_id + 총액 + 품목" 유니크 제약 자체의 위험**
정상 케이스 — *"같은 카트에서 같은 상품을 두 번 주문"* (예: 선물용으로 2번 결제) — 도 차단돼. 게다가 **즉시 구매 플로우는 cart_id가 NULL**일 가능성이 높아. MySQL UNIQUE 인덱스는 NULL을 중복으로 보지 않아서 즉시 구매는 새지. 결국 *매출 손실 최소화* 의도와 정반대로 **정상 매출만 차단되고 중복은 새는** 시나리오가 가능.

**3. "API Gateway 큐잉"이라는 단어의 모호성**
AWS API Gateway에는 **Throttling (Token Bucket 기반 Rate Limit)**은 있지만 "진짜 큐잉"은 없어. 진짜 큐잉은 **API GW → SQS → Lambda/Worker** 비동기 패턴이고 이건 코드 변경이 필수야. 면접관이 *"AWS 어떤 서비스로요?"* 한 마디만 던져도 흔들릴 답. 그리고 Rate Limit도 알고리즘 선택(Token Bucket / Sliding Window / Fixed Window) 언급이 있어야 시니어 답변. 큐잉은 본질적으로 *"지연된 처리"*라 결국 DB로 흘러간다는 함정도 짚어야 해.

**4. 놓친 인프라 옵션 4가지**
- **WAF Rate Rule** — 같은 `user_id` 헤더 / IP로 분당 N회 POST 차단. AWS WAF 콘솔에서 5분 내 적용 가능, 코드 변경 0.
- **ALB / Nginx 레이어 dedup** — `md5(body)` 또는 `(user_id + cart_id)` 키로 N초간 1회만 통과. Nginx `limit_req_zone` + `lua-resty-limit-req` 조합.
- **CloudFront / ALB 셀렉티브 차단** — `/api/orders`만 503으로 응답하고 나머지 트래픽(상품 조회 등)은 보호. 출혈 봉쇄 전략 (bulkhead).
- **Connection Pool 강제 리셋 + max_connections 임시 증대** — `KILL` 또는 풀 재시작으로 행잉 커넥션 스왑.

**5. "온콜 5분 룰" 미충족**
*"내가 지금 알람 받고 5분 안에 무엇을 누르나?"*가 Q3의 본질이야. ALTER TABLE은 5분 안에 누를 수 있는 액션이 아니야. 시니어 답변은 항상 **즉시(0~5min) / 단기(5~30min) / 본 수정(30min+)** 3분류로 분리해. 이번 답변은 즉시 액션과 구조 변경이 한 단락에 섞여 있어서 면접관이 *"어떤 게 가장 먼저예요?"* 물으면 우선순위가 흔들려.

**6. 용어 정밀도: "메타데이터 락" → MDL**
정확히는 **MDL (Metadata Lock)**. MySQL InnoDB에서 `ALTER` 시 short MDL을 잡는데, 진행 중인 트랜잭션이 있으면 풀리지 않고 무한 대기. `pt-osc` / `gh-ost` 같은 도구 이름이 같이 나와야 *"이 사람 운영 DDL 해봤네"*라는 인상을 줘.

### 시니어 모델 답안 (3분류)

**0~5분 — 출혈 봉쇄 (Stop the bleeding)**
- ALB/CloudFront에서 `/api/orders`만 503 일시 차단 **OR** Frontend 주문 버튼 비활성화 + 안내 모달
- 트레이드오프: *완전 차단 → 매출 100% 손실 / 단 5~10분 / 복구 즉시 / 사용자 신뢰 -중*
- 근거: 중복이 매출보다 더 큰 손실(환불·CS·신뢰)을 만들 때 선택. 5분이면 끝.

**5~30분 — 제어된 통과 (Controlled flow)**
- WAF Rate Limit 룰: `user_id` 분당 3회 초과 차단
- Nginx `limit_req_zone`: 동일 `(user_id + body_hash)` 10초간 1회만 통과
- 트레이드오프: *매출 손실 ~5% / 정상 유저도 일부 영향 / 룰 정밀도가 관건 / 복구는 룰 비활성화로 수초*

**30분~수시간 — 구조 보강 (Structural)**
- API GW Throttling 정밀화 (메서드별 limit) — 코드 변경 없음
- DB 풀 / max_connections 임시 증설 + 행잉 커넥션 강제 종료
- (선) 결제 API 비동기 큐(SQS) 분리 → **이건 코드 변경**이라 Q3 범위 밖 / 별도 핫픽스 PR
- 트레이드오프: *근본 해소는 아님 / 시간 벌기 / 본 수정 전까지의 다리(bridge)*

**이후 본 수정 (Permanent fix)**
- 멱등키 구현 (3단계 주제) — Q3에서 다루지 않음

### 트레이드오프 언어 평가 (P0-1 진척)

- **좋아진 것**: *"매출 손실 최소화 vs 차단 강도"*, *"메타락 부작용"* 같은 비교축이 자연스럽게 등장. 한 달 전이면 *"유니크 키 걸면 됨"*에서 끝났을 답.
- **다음 단계**: 트레이드오프를 **(구현 시간 / 매출 영향 % / 복구 난이도)** 3종 세트로 정량화. 시니어 화법 예시 — *"이 옵션은 5분 / 매출 -2% / 룰 OFF로 즉시 복구"*. 다음 답변부터 옵션마다 이 3종 세트를 강제로 붙여보자.

### 면접관 시점

- **합격선 안쪽**: 트레이드오프 의식 + 운영 부작용(MDL) 인식 → 5년차 시스템 디자인 면접 통과 가능선.
- **꼬리질문 위험 2개**:
  1. *"부하 걸린 상태에서 ALTER TABLE을 안전하게 어떻게 거는가?"* → `pt-osc` / `gh-ost` 모르면 막힘
  2. *"API Gateway 큐잉이라고 했는데 정확히 AWS 어떤 서비스인가?"* → Throttling vs SQS 비동기 구분 못 하면 막힘

### 다음 꼬리질문 예고

- **Q3-1**: *"WAF Rate Limit과 Nginx `limit_req_zone`의 차이는? 둘 중 이번 상황에선 어느 게 먼저인가?"*
  → 레이어 위치(엣지 vs 오리진), 룰 적용 속도, 비용·정밀도 트레이드오프를 묻는 질문.
- **Q3-2**: *"주문 API만 503으로 막는 동안 사용자에게 무엇을 보여줘야 하나?"*
  → 단순 차단이 아니라 **UX·신뢰·재시도 유도** 관점. 시니어는 인프라 액션의 *고객 경험 사이드 이펙트*까지 본다.
