# 시스템 디자인 챌린지: 트래픽 100배 폭증 & 중복 주문 장애

> **난이도:** 중급 ~ 중상  
> **예상 소요:** 60~90분  
> **관련 역할:** 백엔드 개발자, 시스템 디자인 면접 대비  
> **날짜:** 2026-04-05

---

## 챌린지 소개

### 시나리오

오늘 오후 2시, 인플루언서가 너희 서비스를 SNS에 올렸다.

**평소 동시 접속 100명이던 서비스에 갑자기 10,000명이 몰려들었다.**

서버는 아직 버티고 있다. CPU 80%, 메모리도 여유가 있다.  
그런데 CS팀에서 슬랙 메시지가 오기 시작한다.

> "고객이 같은 주문이 2~3번 들어왔다고 합니다."  
> "결제는 한 번 했는데 주문 내역에 같은 상품이 여러 개 있대요."  
> "환불 요청이 들어오고 있습니다."

### 서비스 스펙 (가정)

| 항목 | 스펙 |
|------|------|
| 서비스 유형 | 소규모 이커머스 (식품/굿즈) |
| 아키텍처 | 모놀리식 (Monolithic), 단일 서버 |
| 웹 서버 | Nginx + Spring Boot (또는 Node.js/Express) |
| DB | MySQL 단일 인스턴스 (RDS) |
| 캐시 | 없음 (Redis 미도입) |
| 메시지 큐 | 없음 |
| 배포 | 단일 EC2 인스턴스 (Auto Scaling 미설정) |
| 평소 트래픽 | 동시 접속 ~100명, 일 주문 ~500건 |
| 현재 트래픽 | 동시 접속 ~10,000명, 분당 주문 요청 ~2,000건 |
| 주문 API | `POST /api/orders` — 동기 처리, DB 직접 INSERT |

### 핵심 상황 정리

- 서버는 **죽지 않았다** (아직은)
- 하지만 **중복 주문**이 발생하고 있다
- 고객 CS 문의가 **실시간으로 증가** 중
- 너는 이 서비스의 **백엔드 개발자**다

**자, 어떻게 할 거야?**

---

## 1단계: 즉각 대응 — "지금 당장 무엇을 확인하는가?"

CS팀에서 "주문 중복이요!" 라고 슬랙이 왔다. 너는 노트북을 열었다.

**가장 먼저 뭘 확인할 건지, 순서대로 생각해봐.**

### Q1. 중복 주문의 "규모"부터 파악해야 한다. 어떻게 확인할 거야?

DB에서 쿼리를 날린다고 치자. 어떤 기준으로 "중복"을 판별할 수 있어?  
같은 유저가 같은 상품을 샀다고 다 중복은 아니잖아.

> **면접관 코멘트:** "실제 면접에서 이 질문이 나오면, 면접관은 '중복의 정의'를 먼저 물어봅니다. 시간 간격? 동일 상품? 동일 결제 건? 정의를 명확히 하지 않고 솔루션부터 말하면 감점 요인이에요."

### Q2. 지금 이 순간, 서버 상태를 모니터링하려면 어디를 봐야 해?

"CPU 80%"라는 건 알겠는데, 중복 주문과 직접적으로 관련된 지표(Metric)는 뭘까?  
API 응답 시간(Response Time)? DB 커넥션 풀(Connection Pool)? 뭘 봐야 "아, 이래서 중복이 생기는구나"를 알 수 있어?

> **면접관 코멘트:** "여기서 '모니터링 도구가 뭐가 있나요?'라고 역질문하면 좋은 신호입니다. 도구가 없는 상황(우리 시나리오처럼)이라면, 'tail -f로 로그를 보겠다'도 충분히 유효한 답이에요. 중요한 건 '무엇을 보겠다'의 관점이지, 도구 이름이 아닙니다."

### Q3. 중복 주문을 "지금 당장" 막으려면 가장 빠른 방법은 뭘까?

코드를 고치려면 배포가 필요하다. 배포는 시간이 걸린다.  
코드 변경 없이, 인프라 레벨이나 운영 레벨에서 할 수 있는 조치가 있을까?

> **면접관 코멘트:** "이 질문의 의도는 '완벽한 해결책'이 아니라 '긴급 대응 능력'을 보는 겁니다. Rate limiting, 주문 API 앞에 큐를 두는 것, 심지어 일시적으로 주문 버튼을 비활성화하는 것도 답이 될 수 있어요. 트레이드오프(Trade-off)를 설명할 수 있으면 됩니다."

### Q4. 이미 들어간 중복 주문 데이터는 어떻게 처리할 거야?

고객은 이미 CS를 넣었고, 일부는 중복 결제까지 된 상태다.  
데이터 정합성(Data Consistency) 관점에서, 복구 전략을 생각해봐.

> **면접관 코멘트:** "여기서 '결제 취소 API를 호출하겠다'만 말하면 부족합니다. 면접관은 '그 취소가 실패하면?', '부분 취소는?', '고객에게 알림은?' 같은 꼬리 질문을 던집니다. 장애 복구는 단일 액션이 아니라 **파이프라인**이라는 걸 보여줘야 해요."

---

**여기까지 생각해봤으면 나한테 답변해봐.**  
순서가 중요하다. "뭘 먼저 보겠다"는 너의 우선순위를 보여주는 거야.

---

## 2단계: 원인 분석 — "왜 중복 주문이 발생했는가?"

급한 불은 껐다고 치자. 이제 원인을 파야 한다.

"서버가 안 죽었는데 왜 중복이지?" — 이게 핵심이야.  
아래 질문들을 하나씩 생각해봐.

### Q1. 클라이언트(프론트엔드) 쪽에서 중복 요청이 발생할 수 있는 시나리오를 나열해봐.

유저가 "주문하기" 버튼을 눌렀다. 응답이 안 온다. 그럼 유저는 뭘 할까?  
그리고 프론트엔드 코드는 그걸 막고 있을까?

> **면접관 코멘트:** "클라이언트 재시도(Client Retry)는 면접에서 자주 나오는 주제입니다. '버튼 비활성화하면 되죠'만으로는 부족해요. 네트워크 타임아웃(Network Timeout) 후 브라우저의 자동 재시도, SPA에서의 더블 클릭, 모바일 앱의 백그라운드 전환 후 재시도 등 다양한 경로가 있습니다."

### Q2. 서버 쪽에서 "같은 요청을 두 번 처리"할 수 있는 구조적 원인은 뭘까?

현재 주문 API(`POST /api/orders`)는 동기(Synchronous) 처리다.  
요청이 오면 → 재고 확인 → 결제 → DB INSERT 순서로 진행된다.

이 흐름에서 "어느 구간이 느려지면" 중복이 발생할 수 있는지 생각해봐.

> **면접관 코멘트:** "여기서 면접관이 기대하는 답은 '응답 지연(Response Latency)으로 인한 타임아웃 → 클라이언트 재시도 → 서버에서는 두 번 다 처리'라는 흐름입니다. 더 깊이 가면, DB 락(Lock) 경합으로 트랜잭션이 밀리는 시나리오까지 설명할 수 있으면 좋습니다."

### Q3. 네트워크 레이어에서 중복이 발생할 수 있는 경우는?

클라이언트도 한 번만 보냈고, 서버도 정상인데, 중복이 생길 수 있을까?  
Nginx, 로드밸런서(Load Balancer), 또는 TCP 레벨에서 재전송이 일어날 수 있는 상황을 생각해봐.

> **면접관 코멘트:** "이 질문에 답할 수 있으면 '네트워크를 이해하는 개발자'로 평가됩니다. 프록시 타임아웃 후 재시도(Proxy Retry), TCP 재전송(TCP Retransmission), 502 Bad Gateway 후 자동 재시도 등이 키워드입니다."

### Q4. 현재 DB 구조에서 중복 INSERT를 막을 장치가 있는지 확인해봐.

`orders` 테이블에 UNIQUE 제약조건(Unique Constraint)이 걸려 있을까?  
만약 `id`가 auto-increment PK라면, 같은 주문이 여러 row로 들어가는 걸 DB 레벨에서 막을 수 있을까?

> **면접관 코멘트:** "DB 제약조건만으로 멱등성(Idempotency)을 보장할 수 있는지, 없다면 왜 안 되는지를 설명할 수 있으면 중급 이상입니다. '주문 번호를 클라이언트에서 생성하면?'이라는 꼬리 질문도 자주 나옵니다."

---

**여기까지 분석해봤으면 나한테 설명해봐.**  
"왜 중복이 생겼는가"를 한 문장으로 정리할 수 있어야 해.  
그 한 문장이 3단계의 출발점이 된다.

---

## 3단계: 구조 개선 — "내일은 어떻게 만들 것인가?"

원인을 알았으니 이제 구조를 바꿔야 한다.

"다시는 이런 일이 없게" — 근데 현실적으로 가능한 범위에서.  
한 번에 다 못 바꾼다. 우선순위를 정해봐.

### Q1. 멱등성(Idempotency)을 보장하려면 어떤 설계가 필요할까?

같은 요청이 10번 와도 결과는 1번만 반영되어야 한다.  
이걸 어떤 레이어에서, 어떤 키(Key)로 보장할 수 있을까?

힌트: Idempotency Key의 생성 주체는 누구여야 할까? 서버? 클라이언트?

> **면접관 코멘트:** "멱등성 키를 어디에 저장할 것인지(Redis? DB?), TTL은 어떻게 설정할 것인지, 키 충돌(Collision)은 어떻게 처리할 것인지까지 답하면 시니어 수준입니다. Stripe, Toss 같은 결제 API가 어떻게 멱등성을 구현하는지 찾아보면 좋은 레퍼런스가 됩니다."

### Q2. 주문 처리를 비동기(Asynchronous)로 바꾸면 어떤 이점이 있을까?

현재는 `POST /api/orders` → 재고 확인 → 결제 → DB INSERT → 응답.  
이걸 메시지 큐(Message Queue)를 도입해서 바꾼다면, 흐름이 어떻게 달라질까?

힌트: 동기 처리에서 "응답 지연"이 중복의 원인이었다면, 비동기는 그 문제를 어떻게 우회하는 걸까?

> **면접관 코멘트:** "비동기로 바꾸면 '주문 접수 완료'를 먼저 응답하고, 실제 처리는 뒤에서 한다는 건데요 — '그럼 결제 실패하면 어떡하죠?'가 반드시 따라옵니다. 보상 트랜잭션(Compensating Transaction), SAGA 패턴까지 언급할 수 있으면 아주 좋습니다."

### Q3. DB가 병목이었다면, 읽기/쓰기 분리(Read/Write Splitting)를 어떻게 적용할 수 있을까?

현재 MySQL 단일 인스턴스다. 주문이 폭증하면 쓰기(Write) 부하가 급증한다.  
이때 읽기(Read)와 쓰기(Write)를 분리하면 어떤 효과가 있을까?  
그리고 복제 지연(Replication Lag)은 이 시나리오에서 어떤 문제를 일으킬 수 있을까?

> **면접관 코멘트:** "Read Replica를 도입하면 읽기 부하는 줄지만, 쓰기 부하는 그대로입니다. 면접관은 '쓰기 부하를 줄이려면?'이라고 물을 겁니다. 샤딩(Sharding)? 캐싱(Caching)? 큐잉(Queuing)? 각각의 트레이드오프를 설명할 수 있어야 합니다."

### Q4. 트래픽 급증 자체에 대한 방어는 어떻게 설계할 거야?

이번엔 "예고 없이" 트래픽이 100배 뛰었다.  
다음에도 이런 일이 생길 수 있다. 어떻게 대비할 거야?

힌트: Rate Limiting, Circuit Breaker, Auto Scaling, CDN — 이 중에서 이 시나리오에 가장 효과적인 건 뭘까?

> **면접관 코멘트:** "'오토스케일링 걸면 되죠'는 정답이 아닙니다. 스케일링에는 시간이 걸리고(Cold Start), DB는 스케일링되지 않기 때문이죠. 면접관은 '서버를 늘려도 DB가 버텨요?'라고 물을 겁니다. 각 방어 수단이 어떤 레이어를 보호하는지 명확히 구분해야 합니다."

### Q5. 이 모든 개선을 한 번에 할 수는 없다. 우선순위를 정해봐.

시간과 리소스는 제한적이다. 가장 먼저 해야 할 것부터 순서를 매겨봐.  
그리고 그 이유를 설명해봐.

> **면접관 코멘트:** "이 질문이 사실 가장 중요합니다. 기술을 아는 것과, 제약 조건 안에서 결정을 내리는 것은 다른 능력이에요. '이걸 먼저 하는 이유는 ROI가 높아서'라고 말할 수 있으면, 시니어 마인드셋을 보여주는 겁니다."

---

**여기까지 설계해봤으면 나한테 답변해봐.**  
"내일 아침 스프린트 회의에서 팀에게 이 개선안을 설명한다"는 마음으로 정리해봐.

## 참고 이론 키워드

이 챌린지를 풀면서 만나게 되는 CS 이론들이다.  
챌린지를 먼저 풀고, 아래 키워드를 하나씩 파고들어가면 된다.

| 키워드 | 설명 | 관련 단계 |
|--------|------|-----------|
| **멱등성 (Idempotency)** | 같은 연산을 여러 번 수행해도 결과가 동일한 성질 | 2단계, 3단계 |
| **분산 락 (Distributed Lock)** | 여러 서버/프로세스 간 동시 접근을 제어하는 메커니즘 | 2단계, 3단계 |
| **메시지 큐 (Message Queue)** | 비동기 처리를 위한 중간 버퍼, Producer-Consumer 패턴 | 3단계 |
| **SAGA 패턴** | 분산 트랜잭션을 보상 트랜잭션으로 관리하는 패턴 | 3단계 |
| **Rate Limiting** | 단위 시간당 요청 수를 제한하는 기법 | 1단계, 3단계 |
| **Circuit Breaker** | 장애 전파를 막기 위해 호출을 차단하는 패턴 | 3단계 |
| **낙관적/비관적 락 (Optimistic/Pessimistic Lock)** | DB 레벨에서 동시성을 제어하는 두 가지 전략 | 2단계 |
| **CAP 정리 (CAP Theorem)** | 분산 시스템에서 일관성, 가용성, 분할 내성의 트레이드오프 | 3단계 |
| **읽기/쓰기 분리 (CQRS / Read-Write Splitting)** | 읽기와 쓰기 경로를 분리하여 성능을 최적화하는 패턴 | 3단계 |
| **복제 지연 (Replication Lag)** | Primary-Replica 구조에서 데이터 동기화 지연 문제 | 3단계 |

## 최종 아키텍처 다이어그램

### Before: 현재 구조

```
[Client (Browser/App)]
        |
        | POST /api/orders
        v
  +-- [Nginx] --+
  |              |
  v              |
[Spring Boot]    |
  |              |
  | (동기 처리)   |
  | 1. 재고 확인  |
  | 2. 결제 API  |
  | 3. DB INSERT |
  v              |
[MySQL Single]   |
                 |
  * 문제점:      |
  - 응답 지연 → 클라이언트 재시도 → 중복 주문
  - 멱등성 키 없음
  - DB 단일 장애점 (SPOF)
  - 트래픽 제어 장치 없음
```

### After: 개선 구조

```
[Client (Browser/App)]
        |
        | POST /api/orders
        | + Idempotency-Key 헤더
        v
  +-- [Nginx / API Gateway] --+
```

## 마무리

이 챌린지는 "정답"이 있는 문제가 아니야.  
중요한 건 **네가 어떤 순서로 생각하고, 어떤 근거로 결정을 내리는가**다.

실제 면접에서도 마찬가지야.  
면접관은 "이 사람이 장애 상황에서 패닉하지 않고, 체계적으로 사고할 수 있는가"를 본다.

각 단계별로 네 답변을 정리해서 가져와.  
같이 리뷰하면서, 놓친 부분이 있으면 채워나가자.

**답변 준비되면, 1단계부터 시작해봐.**
