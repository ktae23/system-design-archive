## P0 격차 (Critical — 12주에 반드시 해소)

### P0-1. 설계 판단을 **트레이드오프 언어**로 압축해 말하는 힘

> **한 줄 요약**: "Redis 캐시 도입했어요"는 말하는데, "왜 Write Through가 아니라 Cache Aside를 골랐는지"는 말하지 못함.

#### 실무에서 어떻게 드러나나
- **시나리오 A**: 후배가 "여기 캐시 박으면 될 것 같은데요?"라고 물어올 때, "박아보고 RT 보자"로 결정이 넘어감. 나중에 **캐시 무효화 타이밍 이슈**로 데이터 정합성 사고가 터짐
- **시나리오 B**: 아키텍처 리뷰에서 "이 모듈은 왜 이렇게 설계했나요?"라는 질문에 "○○님이 전에 이렇게 했던 것 같아서…"로 답하게 됨. **결정을 스스로 한 게 아니라 빌려온 셈**

#### 면접에서 어떻게 드러나나
- **꼬리질문 1**: "캐시 쓴다고 하셨는데, 쓰기 패턴에서 정합성 어떻게 보장하셨어요?" → "TTL 짧게 잡았습니다"에서 멈춤
- **꼬리질문 2**: "만약 쓰기가 훨씬 많은 서비스라면 캐시 전략 어떻게 달라질까요?" → **말문이 막힘**. 이 순간 면접관은 "경험은 있는데 원리 없이 했구나"로 결론

#### 역학습 경로
> **시스템 디자인 챌린지**: "인기 상품 조회 API — 초당 10만 요청, 재고 업데이트는 초당 100회"
>
> 이 챌린지를 풀다 보면 자연스럽게 꿰어지는 이론:
> - Cache Aside / Write Through / Write Behind 4개 패턴 비교 → **캐시 일관성 모델**
> - Cache Stampede → **Singleflight / Randomized TTL**
> - TTL 선택 → **Eventual Consistency의 허용 윈도우**
> - 재고 차감 → **Race Condition / Distributed Lock (Redis SETNX)**

### P0-2. DB 동시성 시나리오를 **엮어서** 설명하는 힘

> **한 줄 요약**: MVCC, Phantom Read, Next-Key Lock 용어는 안다. 근데 "이 세 개가 한 시나리오에서 어떻게 얽히는지"는 못 풀어냄.

#### 실무에서 어떻게 드러나나
- **시나리오 A**: "배치 돌리면 주문 테이블 전체에 락이 잡히는 것 같아요"라는 제보를 받았을 때, **`SELECT ... FOR UPDATE`의 범위**와 **Gap Lock**이 만드는 간접 블로킹을 추적할 수 없음
- **시나리오 B**: MySQL에서 `REPEATABLE READ`인데 왜 Phantom이 안 나오는지 모르고 넘어감. 이후에 **PostgreSQL로 이관할 때 동일 가정**으로 설계했다가 터짐

#### 면접에서 어떻게 드러나나
- **꼬리질문 1**: "REPEATABLE READ 레벨에서 Phantom이 나옵니까?" → "그렇죠" (엔진 따라 다른데 그 차이를 모름)
- **꼬리질문 2**: "Lost Update와 Phantom Read 구분해주세요. 그리고 각각 어떻게 막죠?" → **섞어서 설명함**. `SELECT FOR UPDATE`만 말하는데, 두 문제의 해결 전략이 본질적으로 다름을 놓침

#### 역학습 경로
> **시스템 디자인 챌린지**: "티켓 예매 시스템 — 공연 1천 석, 오픈 순간 10만 명 동시 접속"
>
> 이 챌린지를 풀다 보면 자연스럽게 꿰어지는 이론:
> - 좌석 중복 예매 방지 → **Lost Update 문제 정의**
> - 비관적 락 vs 낙관적 락 → **`SELECT FOR UPDATE` vs Version Column**
> - 락 범위 최소화 → **Next-Key Lock / Gap Lock 이해**
> - Isolation Level 선택 → **MVCC 스냅샷 시맨틱**
> - 대기열 큐 도입 → **락 경합 → 큐잉으로 시스템 디자인 전환**

---

### P0-3. 시스템 장애의 **원인을 레이어 거슬러** 추적하는 사고

> **한 줄 요약**: "p99 latency가 튄다"는 현상을 보고 **JVM → OS → 커널 → 네트워크**로 훑는 체크리스트가 없음.

#### 실무에서 어떻게 드러나나
- **시나리오 A**: 새벽 2시 p99 응답이 2초로 튐. `top`에서 CPU 사용률은 낮음. 여기서 막힘. 실제로는 **GC pause + thread pool starvation + 외부 API 타임아웃**이 연쇄 발동 중인데, 한 레이어만 보고 "일시적 이슈"로 처리
- **시나리오 B**: "Too many open files" 에러에 단순히 `ulimit` 올려서 대응. 실제 원인은 **커넥션 풀 leak (close() 누락)**이었고, 근본 해결을 못 함

#### 면접에서 어떻게 드러나나
- **꼬리질문 1**: "서비스 latency가 갑자기 튀었을 때 어디부터 보세요?" → "로그 봐요" / "APM 봐요" 정도. **"JVM GC 로그 → OS load avg → TCP retransmit → DB slow query"** 같은 체계적 순서가 없음
- **꼬리질문 2**: "`CLOSE_WAIT`이 쌓이는 상황, 원인 추정해주세요" → "서버가 close 안 해서요" (맞는데, **왜 애플리케이션이 close를 빼먹는지 코드 관점의 원인**까지 가야 함 — try-with-resources 누락, async 콜백에서의 리소스 해제 실패 등)

#### 역학습 경로
> **시스템 디자인 챌린지**: "p99 SLO를 200ms로 보장하는 API 설계 — 내부 의존성 3개, 외부 결제 API 1개"
>
> 이 챌린지를 풀다 보면 자연스럽게 꿰어지는 이론:
> - Fan-out 호출 타임아웃 설계 → **Timeout / Retry / Circuit Breaker**
> - 스레드풀 격리 → **Bulkhead Pattern**
> - latency budget 배분 → **꼬리 지연(tail latency) 이론**
> - 장애 격리 → **Fail-Fast vs Fail-Safe**
> - 모니터링 메트릭 선택 → **USE / RED / Golden Signals 프레임워크**

## P1 격차 (High — 12주에 80% 해소 목표)

### P1-1. 네트워킹의 **애플리케이션 레이어까지 이어지는 이해**

> **한 줄 요약**: TCP/HTTP 용어는 아는데, **Connection Pool, Keep-Alive idle timeout, HTTP/2 multiplexing** 같은 애플리케이션 레이어 맥락과 연결이 안 됨.

#### 실무에서 어떻게 드러나나
- 마이크로서비스 간 gRPC 호출에서 **하나의 채널**이 왜 수천 RPS를 버티는지 몰라서, 서비스마다 채널을 새로 만듦 → 커넥션 폭증
- LB idle timeout이 60초인데 애플리케이션 Keep-Alive가 120초로 잡혀있어, **유휴 커넥션이 LB에서 끊긴 채 애플리케이션은 "살아있다고 믿는 상태"**가 되어 첫 요청마다 실패

#### 면접에서 어떻게 드러나나
- "HTTP/1.1과 HTTP/2 차이?" → "HTTP/2는 multiplexing 된다" → **"그럼 gRPC는 왜 HTTP/2 위에서 도는 거죠?"**에서 막힘
- "커넥션 풀 사이즈는 어떻게 정하세요?" → **경험적 숫자**만 나오고 "동시 요청 수 × 평균 응답 시간 / 1초" 같은 **Little's Law 기반 계산**이 안 나옴

#### 역학습 경로
> **시스템 디자인 챌린지**: "마이크로서비스 10개, 서비스 간 통신 빈도 높음 — 통신 레이어 설계"
>
> 자연스럽게 꿰어지는 이론:
> - HTTP/1.1 vs HTTP/2 → **Head-of-Line Blocking**
> - gRPC 채널 구조 → **HTTP/2 multiplexing + stream**
> - Connection Pool 사이징 → **Little's Law**
> - Keep-Alive / Idle Timeout 정렬 → **LB / Proxy와의 계약**

---

### P1-2. OS 레이어의 **성능/장애 트리거 지점 감각**

> **한 줄 요약**: 용어(Fork, Thread, Blocking)는 구분하는데, **이게 실제 언제 성능 절벽을 만드는가**를 체감으로 못 가지고 있음.

#### 실무에서 어떻게 드러나나
- Java 애플리케이션의 스레드풀 크기를 **코어 수와 연결 짓지 않고** 관습적으로 200개 잡음. I/O bound vs CPU bound에 따라 공식이 다르다는 걸 모름
- JVM이 컨테이너 환경에서 **메모리 제한을 인식 못 하고** OOM Kill 당한 경험이 있는데, **cgroup과 JVM의 관계**를 몰라서 재발

#### 면접에서 어떻게 드러나나
- "스레드풀 크기 정하는 기준?" → "경험상…" → **"CPU bound면 코어 수, I/O bound면 Amdahl 공식"** 같은 근거 없음
- "Redis가 싱글스레드인데 왜 빠를까요?" → "빠르니까요" 수준에서 멈춤. **"이벤트 루프 + epoll + in-memory"**의 조합을 설명 못 함

#### 역학습 경로
> **시스템 디자인 챌린지**: "초당 5만 RPS 받는 게이트웨이 설계"
>
> 자연스럽게 꿰어지는 이론:
> - 스레드 모델 선택 → **Thread-per-connection vs Event Loop vs Coroutine**
> - 커널 네트워크 스택 → **epoll / kqueue / io_uring**
> - 컨테이너 리소스 제약 → **cgroup / JVM 튜닝**
> - Context switch 비용 → **이론적 상한과 C10K/C10M 문제**

---

### P1-3. 실무 자료구조 감각 — "Redis 자료형과 자료구조 매핑"

> **한 줄 요약**: 알고리즘 문제를 못 푸는 건 괜찮지만, **Redis/DB 설계에서 자료구조를 근거로 선택**하는 힘이 약함.

#### 실무에서 어떻게 드러나나
- 실시간 랭킹 만들 때 Redis ZSET을 쓰긴 하는데, **내부가 Skip List라서 `ZRANGE`가 O(log N + M)**이라는 근거로 "백만 건 넘어가면 Top 100 조회 괜찮겠다"를 말하지 못함 → 과도한 샤딩이나 불필요한 캐싱을 추가
- "최근 본 상품 10개"를 RDB `ORDER BY created_at DESC LIMIT 10`으로 구현 → Redis List + LTRIM이면 O(1)인데 모름

#### 면접에서 어떻게 드러나나
- "실시간 랭킹 설계해주세요" → Redis ZSET까지 나오는데, **"내부가 Skip List라서…"**까지 이어지는 경우가 드묾
- "중복 방문자 수 세야 하는데 메모리가 많지 않아요" → **HyperLogLog 떠올리기 어려움**
- "대량 데이터에서 특정 원소 존재 여부 빠르게 확인" → Set만 말하고, **Bloom Filter가 안 나옴**

#### 역학습 경로
> **시스템 디자인 챌린지**: "피드 서비스 — 타임라인, 실시간 랭킹, 읽음 처리, 중복 알림 방지"
>
> 자연스럽게 꿰어지는 이론:
> - 타임라인 저장 → **Redis List / Sorted Set 선택**
> - 실시간 랭킹 → **Skip List 복잡도**
> - 읽음 여부 → **Bitmap / HyperLogLog**
> - 중복 알림 방지 → **Bloom Filter**
> - 캐시 만료 → **LRU / LFU 자료구조**

---

### P1-4. 분산 시스템 **장애 시나리오와 방어 패턴**

> **한 줄 요약**: 단일 서비스 개발은 익숙한데, **서비스 간 연결이 만드는 "실패의 확산"**에 대한 대비가 체계적이지 못함.

#### 실무에서 어떻게 드러나나
- 결제 API 호출 실패 시 **그냥 retry** — retry 폭탄이 외부 API를 죽이고, 우리 서비스도 같이 죽는 "cascade failure" 경험
- 이벤트 기반으로 주문 → 포인트 적립 시, **중복 이벤트 처리**가 없어서 포인트가 두 번 적립되는 사고

#### 면접에서 어떻게 드러나나
- "외부 API 호출 실패하면 어떻게 처리하셨나요?" → "retry 걸었어요" → **"얼마나? 어떤 조건에서 중단? Circuit Breaker는?"**에서 막힘
- "Kafka 컨슈머가 메시지를 처리했는데 커밋 전에 죽으면?" → **idempotent consumer의 필요성**까지 못 가고 "다시 처리되겠죠"로 끝남

#### 역학습 경로
> **시스템 디자인 챌린지**: "주문 → 결제 → 재고 → 배송" 4단계 분산 처리 설계
>
> 자연스럽게 꿰어지는 이론:
> - 분산 트랜잭션 불가능성 → **Saga Pattern**
> - 메시지 중복 → **Idempotency Key / Outbox Pattern**
> - retry 정책 → **Exponential Backoff + Jitter**
> - 장애 격리 → **Circuit Breaker / Bulkhead**
> - CDC → **DB ↔ Event 동기화**

## 역학습 경로 통합 뷰: 시스템 디자인 챌린지 → 이론 매핑

각 P0/P1 격차에 매핑된 챌린지들을 **12주에 배치할 수 있는 순서**로 정리.

| 챌린지 | 풀리는 격차 | 드러나는 핵심 이론 |
|---|---|---|
| 1. 인기 상품 조회 API (10만 RPS) | P0-1, P1-3 | Cache Aside/Write Through, Cache Stampede, Redis 자료형 |
| 2. 티켓 예매 시스템 (동시성) | P0-2 | MVCC, Lock, Isolation Level, 낙관/비관 락 |
| 3. p99 SLO 200ms API | P0-3, P1-2 | Timeout/Circuit Breaker, Bulkhead, Thread Model |
| 4. 마이크로서비스 통신 | P1-1 | HTTP/2, gRPC, Little's Law, Connection Pool |
| 5. 피드 서비스 (타임라인+랭킹) | P1-3 | Skip List, Bloom Filter, HyperLogLog, LRU |
| 6. 주문→결제→재고→배송 | P1-4 | Saga, Idempotency, Outbox, CDC |

**원칙**: 이론을 먼저 읽고 챌린지로 검증하는 게 아니라, **챌린지 설계 중 "막히는 지점"에서 이론을 정확히 필요한 깊이까지만 들이는다**. 이게 역학습의 핵심이다.

## 12주 커리큘럼 힌트 — 방향성 제시

### 전체 구조 제안: 3단계 × 4주

#### Phase 1 (Week 1~4): **기초 단단히 + 설계 언어 장착**
- 목표: **트레이드오프 언어**와 **DB 동시성**을 확실히 잡기 (P0-1, P0-2 공략)
- 챌린지: 인기 상품 조회 API, 티켓 예매 시스템
- 산출물: 설계 문서 작성 → "나의 판단 근거를 글로 압축"하는 훈련 시작

#### Phase 2 (Week 5~8): **레이어 거슬러 올라가기 + 분산 세계 입문**
- 목표: **OS/네트워크 레이어 진단 능력** + **분산 시스템 방어 패턴** (P0-3, P1-1, P1-2, P1-4 공략)
- 챌린지: p99 SLO API, 마이크로서비스 통신, 주문/결제 Saga
- 산출물: 장애 시나리오 플레이북, 아키텍처 다이어그램 3개

#### Phase 3 (Week 9~12): **실무 자료구조 + 면접 시뮬레이션 + 이력서/포트폴리오**
- 목표: **실무 자료구조 감각 완성** + **면접 대응력** (P1-3, P2-1 정리 + 이직 준비)
- 챌린지: 피드 서비스, 이력서 기반 mock interview
- 산출물: 시스템 디자인 포트폴리오 (5개 설계), 이력서 v2, 예상 꼬리질문 대응 드릴

### 배치 원칙

1. **DB 동시성을 Phase 1 초반에 배치** — 이 사람의 가장 단단한 영역이어서 **"성공 경험으로 시작"**하는 게 동기부여에 유리
2. **OS/네트워크는 단독 학습으로 배치하지 않음** — 시스템 디자인 챌린지의 **"왜 이렇게 설계했나?"** 꼬리질문에서 발굴
3. **알고리즘은 백그라운드 과제로** — 매주 LeetCode 2~3문제, 단독 주간 할당 없음
4. **이력서/면접 준비는 Phase 3에 집중하되, Phase 1~2에서 "설계 문서 작성" 습관을 미리 들여놓음** — 설계 문서가 곧 이력서의 기술 블럭이 됨
5. **매주 1개 챌린지 + 1개 회고** — 회고에서 **"무엇을 몰랐는지"**를 명시적으로 기록 → 다음 주 이론 학습의 방향이 됨

### 주의할 함정

- **이 사람은 "공부형"이 아니라 "실행형"이다** (자가진단 톤에서 보인다). 그래서 **이론 주간을 오래 잡지 말 것**. 챌린지 안에서 이론이 소비되게 할 것
- **"다 알아야 한다"는 압박을 주지 말 것** — 5년차가 모르는 게 많다는 건 당연하다. **"어느 레벨까지는 가야 한다"**를 명확히
- **DB 중급 중위를 **첫 주 성공 경험**으로 활용**: "이 영역은 당신이 이미 잘합니다, 여기서 출발해서 깊이를 더합시다"

## 요약 한 문장

> **이 사람은 "경험은 있는데 언어가 부족한" 5년차**다. 12주 커리큘럼의 중심축은 **시스템 디자인 챌린지를 통해 경험을 "트레이드오프 언어"로 압축하는 훈련**이어야 한다. 이론은 챌린지에 부속되는 형태로만 들여오고, 독립 주간으로 잡지 않는다.
